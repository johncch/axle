# Migrating to Axle 0.22.0

0.22.0 focuses on provider-managed web search and source rendering:

- OpenRouter server tools through the Chat Completions provider
- ordered citation parts for unanchored provider sources
- web-search request counts in usage stats when available
- warning diagnostics for unexpected unanchored citations from native providers

Most applications only need changes if they render turns exhaustively or use
OpenRouter web search.

---

## Citation Parts

0.21.0 introduced citations attached to text parts:

```ts
if (part.type === "text") {
  renderText(part.text, part.citations);
}
```

0.22.0 adds a separate ordered part for citations that are not anchored to a
specific text span.

```ts
interface CitationPart {
  type: "citation";
  citations: Citation[];
  providerMetadata?: Record<string, unknown>;
}
```

This applies to both model-facing assistant messages and renderable turns:

- `ContentPartCitation`
- `CitationPart`

### Why

Some providers emit citations as source lists for the whole assistant message,
not as spans inside a specific text block. OpenRouter web search in the Chat
Completions API does this: citations can arrive as standalone
`delta.annotations[]` chunks, and their `start_index` / `end_index` can be
`0` / `0`.

Axle no longer forces those citations onto whichever text or thinking part
happens to be open. It preserves them as ordered `citation` parts.

### Before

Renderers could reasonably assume that all citations lived on text parts:

```ts
for (const part of turn.parts) {
  if (part.type === "text") {
    renderText(part.text);
    renderInlineCitations(part.citations ?? []);
  }
}
```

### After

Renderers should also handle `citation` parts:

```ts
for (const part of turn.parts) {
  if (part.type === "text") {
    renderText(part.text);
    renderInlineCitations(part.citations ?? []);
  }

  if (part.type === "citation") {
    renderSources(part.citations);
  }
}
```

Text-span citations still use `TextPart.citations`. The new `citation` part is
for unanchored sources where there is no honest text owner.

## Exhaustive Turn Rendering

If your code switches exhaustively on `TurnPart["type"]`, add a `citation`
case.

### Before

```ts
switch (part.type) {
  case "text":
  case "thinking":
  case "file":
  case "action":
    renderPart(part);
    break;
}
```

### After

```ts
switch (part.type) {
  case "text":
  case "citation":
  case "thinking":
  case "file":
  case "action":
    renderPart(part);
    break;
}
```

Likewise, code switching on model-facing `ContentPart["type"]` should add
`"citation"`.

## Stream Events

Low-level stream consumers can now receive an ordered citation event:

```ts
{
  type: "citation";
  index: number;
  citations: Citation[];
  providerMetadata?: Record<string, unknown>;
}
```

`TurnEventBuilder` converts this into a normal `part:start` / `part:end` pair
for a `CitationPart`. If you use `TurnAccumulator`, no reducer changes are
needed beyond rendering the new part type.

Existing span-anchored citation events are unchanged:

```ts
{
  type: "text:citation";
  index: number;
  citation: Citation;
  citations: Citation[];
}
```

## OpenRouter Web Search Server Tool

The Chat Completions provider can now map Axle provider tools to OpenRouter
server tools. Opt into the OpenRouter mapping at provider construction:

```ts
import { Agent, chatCompletions } from "@fifthrevision/axle";

const provider = chatCompletions("https://openrouter.ai/api/v1", {
  apiKey: process.env.OPENROUTER_API_KEY,
  providerToolVendor: "openrouter",
});
```

Then use Axle's provider-neutral tool identity:

```ts
const agent = new Agent({
  provider,
  model: "openai/gpt-4o-search-preview",
  providerTools: [{ type: "provider", name: "web_search" }],
});
```

With OpenRouter-specific config:

```ts
providerTools: [
  {
    type: "provider",
    name: "web_search",
    config: { max_results: 3 },
  },
];
```

Axle sends this as an OpenRouter server tool:

```json
{
  "type": "openrouter:web_search",
  "parameters": {
    "max_results": 3
  }
}
```

Only `type: "provider"` and `name: "web_search"` are portable. `config` is a
provider-specific passthrough: field names and placement differ across
providers.

Examples:

| Provider                     | Example config fields                                |
| ---------------------------- | ---------------------------------------------------- |
| OpenAI                       | `search_context_size`, `user_location`               |
| Anthropic                    | `max_uses`, `allowed_domains`, `blocked_domains`     |
| OpenRouter server tool       | `max_results`, `allowed_domains`, `excluded_domains` |
| OpenRouter deprecated plugin | `include_domains`                                    |

Function tools and OpenRouter server tools share the same Chat Completions
`tools` array. Generic OpenAI-compatible Chat Completions providers still warn
and drop `providerTools` unless `providerToolVendor: "openrouter"` is set.

## Web Search Billing

`Stats` remains token-focused. Provider-managed web search can incur off-token
fees, but providers do not expose a portable per-turn search count or cost field.
Axle surfaces search artifacts through provider-tool parts and citations; hosts
that estimate billing should treat web-search fees as provider-specific.

---

## OpenRouter Plugin Path

The OpenRouter plugin path is unchanged:

```ts
providerOptions: {
  plugins: [{ id: "web" }],
}
```

Prefer `providerTools: [{ type: "provider", name: "web_search" }]` when you
want model-driven web search, because the server tool lets the model decide
whether and when to search.

---

## Citation Diagnostics

Axle now logs `console.warn` when native providers emit citation data that does
not line up with their expected text-anchored format:

- OpenAI Responses annotation without a resolved output text part
- Anthropic citation delta outside a text block
- Gemini citation metadata without a part index

These warnings are diagnostic only. Existing native-provider citation routing is
unchanged: anchored citations still attach to text parts.

If you see one of these warnings in production, capture the provider payload.
It may indicate that the provider has started emitting unanchored sources that
should be routed to `CitationPart`.
