# Migrating to Axle 0.23.0

0.23.0 reworks the observability interface around a single tracing seam and
brings the non-streaming path to parity with streaming:

- `observability.tracer` is replaced by `observability.trace`, which accepts a
  `Tracer` **or** a `Span`
- `log` and `trace` are mutually exclusive; compose writers for multiple sinks
- `TracingContext` is renamed to `Span`
- `SpanData` timestamps are now epoch milliseconds
- `new Tracer({ minLevel, writers })` ergonomic constructor
- streaming and non-streaming runs log identical turn content (text, thinking,
  provider-tools, citations)

Most applications only need changes if they configure `observability`, import
`TracingContext`, or implement a custom `TraceWriter`.

---

## `observability.tracer` → `observability.trace`

The bring-your-own field is renamed and its type is widened. It now takes
anything you can open a span under — a `Tracer` (each send is its own root
trace) or a `Span` (sends nest under it).

### Before

```ts
new Agent({
  provider,
  model,
  observability: { tracer: myTracer },
});
```

### After

```ts
new Agent({
  provider,
  model,
  observability: { trace: myTracer },
});
```

To nest an agent's spans under an existing span — e.g. a CLI job span — pass the
span instead:

```ts
const job = tracer.startSpan("job", { type: "workflow" });
new Agent({ provider, model, observability: { trace: job } });
```

### Why

The agent only needs `startSpan` to attach `agent.send`. Both `Tracer` and
`Span` provide it, so one field covers "trace this run as a root" and "nest this
run under my span" without a second option. Axle attaches its spans to whatever
you pass but never ends or flushes it — that stays the caller's job.

The `log`-only path is unchanged:

```ts
observability: { log: (entry) => logger[entry.level](entry.message, entry.fields) }
```

---

## `log` and `trace` Are Mutually Exclusive

Previously, passing both a `tracer` and `log` attached a `LogWriter` to your
tracer. That combo is removed: `log` is sugar for "Axle creates and owns a
tracer with a `LogWriter`," and once you bring your own tracer you own its
writers. If both are set, `trace` wins and Axle emits a `console.warn`.

To send spans to more than one sink (e.g. structured logs **and** an OTel
exporter), compose the writers on one tracer and pass it as `trace`.

### Before

```ts
const tracer = new Tracer();
tracer.addWriter(new OTelWriter());

observability: {
  tracer, // Axle also attached a LogWriter from `log`
  log: myLog,
}
```

### After

```ts
import { Tracer, LogWriter } from "@fifthrevision/axle";

const tracer = new Tracer({
  writers: [new OTelWriter(), new LogWriter(myLog)],
});

new Agent({ provider, model, observability: { trace: tracer } });
```

`LogWriter` is exported, so existing log output is byte-for-byte preserved — you
are just placing it next to your other writers.

---

## `TracingContext` → `Span`

The span interface is renamed. It is the same shape (`startSpan`, `end`,
`info`/`debug`/…, `setAttribute(s)`, `setResult`).

### Before

```ts
import type { TracingContext } from "@fifthrevision/axle";

function instrument(span: TracingContext) { /* … */ }
```

### After

```ts
import type { Span } from "@fifthrevision/axle";

function instrument(span: Span) { /* … */ }
```

No behavior change — rename the type at import and usage sites.

---

## Threaded `tracer` Parameter → `span`

The active span passed into provider, tool, memory, and MCP calls was named
`tracer`, but it is a `Span` — you call `.startSpan` / `.info` / `.end` /
`.setResult` on it, none of which a `Tracer` has. It is renamed `span` to match
its type and to stay distinct from the `Tracer | Span` config you hand the agent
as `observability.trace`.

This affects code that reads the span from a context, or passes it into
`generate` / `stream` directly.

### Before

```ts
const tool: ExecutableTool = {
  // …
  async execute(params, ctx) {
    ctx.tracer?.setAttribute("detail", summarize(params));
  },
};

await generate({ provider, model, messages, tracer: span });
```

### After

```ts
const tool: ExecutableTool = {
  // …
  async execute(params, ctx) {
    ctx.span?.setAttribute("detail", summarize(params));
  },
};

await generate({ provider, model, messages, span });
```

The rename is mechanical and applies everywhere the span is threaded:

- `ToolContext.tracer` → `ToolContext.span`
- `MemoryContext.tracer` → `MemoryContext.span`
- `GenerateParams.tracer` and the `stream` params `tracer` → `span`
- MCP `connect` / `listTools` / `close` options `tracer` → `span`

`observability.trace` on the `Agent` is unaffected — that is the config knob
(`Tracer | Span`), not the threaded span.

## `SpanData` Timestamps Are Epoch Milliseconds

`SpanData.startTime` / `SpanData.endTime` and `SpanEvent.timestamp` were
monotonic values from `performance.now()` (milliseconds since process start).
They are now **epoch milliseconds** — derived from the monotonic clock via
`performance.timeOrigin`, so absolute placement works while durations stay
exact.

This only affects custom `TraceWriter` implementations that read timestamps.

### Before

```ts
onSpanEnd(span: SpanData) {
  // startTime was relative; turning it into wall-clock needed timeOrigin
  const startedAt = new Date(performance.timeOrigin + span.startTime);
}
```

### After

```ts
onSpanEnd(span: SpanData) {
  const startedAt = new Date(span.startTime);          // already epoch ms
  const durationMs = span.endTime! - span.startTime;   // unchanged — still exact
}
```

### Why

Epoch milliseconds are the most widely translatable internal format: every sink
(OTel `startTime` accepts a `number` as epoch ms, `Date`, log timestamps) reads
them directly, while `endTime - startTime` cancels `timeOrigin` and remains a
precise duration. Writers no longer add `timeOrigin` themselves.

---

## `new Tracer({ minLevel, writers })`

The `Tracer` constructor takes an optional options bag so the common setup reads
as one statement. The imperative API (`addWriter`, `removeWriter`, `minLevel`)
is unchanged.

### Before

```ts
const tracer = new Tracer();
tracer.minLevel = "debug";
tracer.addWriter(new SimpleWriter());
```

### After

```ts
const tracer = new Tracer({ minLevel: "debug", writers: [new SimpleWriter()] });
```

`new Tracer()` with no arguments still works.

---

## Turn Content Logging

Both the streaming and non-streaming (`generate`) paths now log a turn's content
onto its span: `text` (info, truncated), `thinking` (debug), provider-tools, and
citations. Previously only the streaming path did this, and provider-managed web
search — which OpenRouter surfaces as citation parts rather than a tool call —
left no trace.

Per searching turn you will now see a `citations` event:

```json
{ "level": "info", "name": "citations", "count": 6, "sources": [{ "type": "web", "title": "…", "url": "https://…" }] }
```

`count` is every citation; `sources` is deduplicated by URL and capped (the full
list logs at `debug` when truncated), and `citationCount` is set as a span
attribute. This is a logging change only — no API surface is affected, but `info`
output is more detailed for turns that ground on sources or call provider tools.
