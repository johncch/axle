# Axle 0.20.0 Notes: Core/CLI Boundary And Rehydration

This is a temporary planning note for follow-up work after the 0.19.0
turn-event / annotation / accumulator release.

## Current Conclusion

0.20.0 should focus on the core/CLI boundary and rehydration model.

0.19.0 shipped the turn-event / annotation / accumulator release. The next
boundary to define is how an agent is described, serialized, reconstructed, and
owned by either core or CLI code.

## Proposed 0.20.0 Theme

Core/CLI boundary and serializable agent definitions.

Potential scope:

- split core library and CLI into separate packages
- keep them in the same repo initially
- move CLI-specific YAML/job loading into the CLI package
- move batch runner, ledger, dotenv loading, and CLI-only tool wiring into CLI
- keep core Axle focused on runtime concepts: `Agent`, `stream`, `generate`,
  providers, messages, turns, turn events, accumulators, tool interfaces, MCP
  abstraction
- define a serializable `AgentDefinition` and `AgentSession`
- use `.env` / environment variable references for secrets instead of provider
  secrets files

## Package Shape

Prefer a workspace split before a separate repo:

```txt
packages/
  axle/        # core library/runtime
  axle-cli/    # CLI app, YAML runner, local workflow tools
```

Core package:

- `Agent`
- `generate`, `stream`
- provider interfaces and provider implementations
- `AxleMessage`
- `Turn`, `TurnEvent`, `TurnAccumulator`
- tool interfaces
- MCP abstraction, if it remains library-level
- `AgentSession`
- `AgentDefinition`
- `createAgentConfig()`

CLI package:

- `src/cli.ts`
- YAML job loading and schemas
- `.env` loading
- config/secrets migration
- provider config resolution from env/YAML
- batch runner
- ledger
- built-in local workflow tools such as exec/read/write/patch/brave, unless
  those move to a separate tools package
- examples/jobs
- JSON schema files for job YAML

Use workspace dependency during development:

```json
{
  "dependencies": {
    "@fifthrevision/axle": "workspace:*"
  }
}
```

This preserves the current dogfooding property:

```txt
repo code + CLI jobs + runtime APIs stay in lockstep
```

while still creating a real package boundary.

## Why Not Separate Repo Yet?

A separate repo improves purity but adds release coordination. Axle currently
uses `.job` files for internal bookkeeping and development workflows such as
models, changelog, and eval-style runs. Keeping CLI and core in one repo means
those jobs can run against the current workspace implementation.

Separate package, same repo gives most of the boundary benefit without the
coordination tax.

## Rehydration Model

Break rehydration into two serializable concepts:

```ts
interface SavedAgent<TAnnotation extends Annotation = Annotation> {
  definition: AgentDefinition;
  session: AgentSession<TAnnotation>;
}
```

`AgentDefinition` is the reconstructable recipe. It should be serializable and
close to the YAML format, but not necessarily identical to today's job format.

`AgentSession` is the mutable continuation/render state.

`Agent` owns a stable `sessionId`, generated by default and overridable by
hosts that already have a session identity. Memory receives that `sessionId`,
but memory-specific partitioning such as user/workspace scope belongs to the
memory implementation or memory config, not `AgentConfig` or
`AgentDefinition`.

The core continuation API is:

```ts
const session = agent.snapshot();
agent.restore(session);
```

### AgentDefinition

Implemented in core as:

- `AgentDefinition`
- `AgentDefinitionResolver`
- `SavedAgent`
- `createAgentConfig()`

Sketch:

```ts
interface AgentDefinition {
  version: 1;
  name?: string;

  provider: ProviderDefinition;
  model?: string;
  system?: string;

  request?: AgentDefinitionRequestOptions;

  tools?: ToolDefinitionRef[];
  providerTools?: ProviderToolDefinitionRef[];
  mcps?: MCPConfig[];
}
```

This is not executable by itself. The consumer supplies a resolver for live
objects:

```ts
type AgentDefinitionResolver = (
  definition: AgentDefinition,
) => MaybePromise<ResolvedAgentDefinition>;

interface ResolvedAgentDefinition {
  provider: AIProvider;
  model?: string;
  tools?: ExecutableTool[];
  providerTools?: ProviderTool[];
  mcps?: MCP[];
}
```

The host-mediated reconstruction flow:

```ts
const config = await createAgentConfig(saved.definition, resolver);
const agent = new Agent(config);
agent.restore(saved.session);
```

`AgentDefinition` maps to `AgentConfig`, but it is not `AgentConfig`.
`AgentConfig` contains runtime objects. `AgentDefinition` contains serializable
data and references that a host resolves into runtime objects.

### AgentSession

Sketch:

```ts
interface AgentSession<TAnnotation extends Annotation = Annotation> {
  version: 1;
  sessionId: string;
  messages: AxleMessage[];
  turns?: Turn<TAnnotation>[];
  sessionAnnotations?: TAnnotation[];
}
```

Required for model continuation:

- `messages` / `history.log`

Required for exact UI restore:

- `turns`
- `sessionAnnotations`

If `messages` is gone, continuation is impossible. If `turns` are gone, exact UI
restore is impossible, though a rough view could be reconstructed from messages.

## YAML Relationship

Current `JobConfig` mixes reusable agent recipe and invocation concerns:

Reusable agent-ish fields:

- `name`
- `provider`
- `tools`
- `provider_tools`
- `mcps`

Invocation/run fields:

- `task`
- `files`
- `batch`

The CLI maps `JobConfig` directly into runtime construction:

- agent-shaped fields are converted into an internal `AgentDefinition` and then
  materialized into `AgentConfig`;
- invocation fields (`task`, `files`, `batch`) are passed directly to the CLI
  runners;
- memory, file resolution, tracing, stores, transport, and batch execution are
  harness concerns layered around the core agent definition.

Existing YAML remains accepted without adding a separate internal job shape.
The CLI requires an explicit `--job` path and only accepts YAML job files.

## Secrets Direction

Move away from provider secrets files. Prefer `.env` and environment variable
references.

Recommended YAML:

```yaml
provider:
  type: openai
model: gpt-4.1
```

or, after normalizing to `AgentDefinition`:

```yaml
provider:
  type: openai
  config:
    apiKeyEnv: OPENAI_API_KEY
model: gpt-4.1
```

Resolver reads standard env vars:

- `OPENAI_API_KEY`
- `ANTHROPIC_API_KEY`
- `GEMINI_API_KEY`
- `CHATCOMPLETIONS_BASE_URL`
- `CHATCOMPLETIONS_MODEL`
- `CHATCOMPLETIONS_API_KEY`
- `BRAVE_API_KEY`

Support explicit env references:

```yaml
provider:
  type: openai
  config:
    apiKeyEnv: CUSTOM_OPENAI_KEY
model: gpt-4.1
```

Literal secrets in YAML should be discouraged or treated as an escape hatch,
not the happy path. `axle.config.*` is no longer part of the CLI config model;
provider-level keys in the job file override environment variables.

CLI should load `.env`. Core library should not load `.env`; library consumers
can choose dotenv, platform secrets, workers bindings, etc.

## Open Questions

- Should built-in local tools stay in core, move to CLI, or move to a third
  package?
- Should MCP remain core-level or become CLI/host-level?
- Should `Agent.fromSaved(saved, resolver)` exist as a convenience API, or only
  lower-level `new Agent(config)` plus `agent.restore(session)`?
