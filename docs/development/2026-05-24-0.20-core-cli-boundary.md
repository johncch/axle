# Axle 0.20.0 Notes: Core/CLI Boundary And Rehydration

This is a temporary planning note for follow-up work after the 0.19.0
turn-event / annotation / accumulator release.

## Current Conclusion

Do not add the CLI/package split to 0.19.0.

0.19.0 should stay focused on:

- `AgentEvent` -> `TurnEvent`
- `TurnAccumulator`
- annotations
- `TurnEventBuilder`
- presentation-state cleanup
- migration docs for the event model

The CLI/package split should be a 0.20.0 task because it is a different class
of breaking change: package architecture, dependency boundaries, exports,
release flow, YAML ownership, and secret handling.

## Proposed 0.20.0 Theme

Core/CLI boundary and serializable agent specs.

Potential scope:

- split core library and CLI into separate packages
- keep them in the same repo initially
- move CLI-specific YAML/job loading into the CLI package
- move batch runner, ledger, dotenv loading, and CLI-only tool wiring into CLI
- keep core Axle focused on runtime concepts: `Agent`, `stream`, `generate`,
  providers, messages, turns, turn events, accumulators, tool interfaces, MCP
  abstraction
- define a serializable `AgentSpec` and `AgentStateSnapshot`
- use `.env` / environment variable references for secrets instead of provider
  secrets files

## Package Shape

Prefer a workspace split before a separate repo:

```txt
packages/
  axle/        # core library/runtime
  axle-cli/    # CLI app, YAML runner, local workflow tools
```

Core package:

- `Agent`
- `generate`, `stream`
- provider interfaces and provider implementations
- `AxleMessage`
- `Turn`, `TurnEvent`, `TurnAccumulator`
- tool interfaces
- MCP abstraction, if it remains library-level
- eventual `AgentStateSnapshot` type
- maybe eventual `AgentSpec` type

CLI package:

- `src/cli.ts`
- YAML job loading and schemas
- `.env` loading
- config/secrets migration
- provider config resolution from env/YAML
- batch runner
- ledger
- built-in local workflow tools such as exec/read/write/patch/brave, unless
  those move to a separate tools package
- examples/jobs
- JSON schema files for job YAML

Use workspace dependency during development:

```json
{
  "dependencies": {
    "@fifthrevision/axle": "workspace:*"
  }
}
```

This preserves the current dogfooding property:

```txt
repo code + CLI jobs + runtime APIs stay in lockstep
```

while still creating a real package boundary.

## Why Not Separate Repo Yet?

A separate repo improves purity but adds release coordination. Axle currently
uses `.job` files for internal bookkeeping and development workflows such as
models, changelog, and eval-style runs. Keeping CLI and core in one repo means
those jobs can run against the current workspace implementation.

Separate package, same repo gives most of the boundary benefit without the
coordination tax.

## Rehydration Model

Break rehydration into two serializable concepts:

```ts
interface SavedAgent<TAnnotation extends Annotation = Annotation> {
  spec: AgentSpec;
  state: AgentStateSnapshot<TAnnotation>;
}
```

`AgentSpec` is the reconstructable recipe. It should be serializable and close
to the YAML format, but not necessarily identical to today's job format.

`AgentStateSnapshot` is the mutable continuation/render state.

### AgentSpec

Sketch:

```ts
interface AgentSpec {
  version: 1;
  name?: string;

  provider: ProviderSpec;
  model?: string;
  system?: string;

  requestOptions?: Omit<AxleModelRequestOptions, "signal">;

  tools?: ToolUseSpec[];
  providerTools?: ProviderTool[];
  mcps?: MCPConfig[];

  memory?: MemorySpec;
  scope?: Record<string, string>;

  fileResolver?: FileResolverSpec;
}
```

This is not executable by itself. The consumer supplies a resolver for live
objects:

```ts
interface AgentSpecResolver {
  provider(spec: ProviderSpec): AIProvider;
  tools?(specs: ToolUseSpec[]): ExecutableTool[];
  memory?(spec?: MemorySpec): AgentMemory | undefined;
  fileResolver?(spec?: FileResolverSpec): FileResolver | undefined;
}
```

The host-mediated reconstruction flow:

```ts
const config = resolveAgentSpec(spec, resolver);
const agent = new Agent(config);
agent.restoreState(state);
```

### AgentStateSnapshot

Sketch:

```ts
interface AgentStateSnapshot<TAnnotation extends Annotation = Annotation> {
  version: 1;
  messages: AxleMessage[];
  turns?: Turn<TAnnotation>[];
  sessionAnnotations?: TAnnotation[];
}
```

Required for model continuation:

- `messages` / `history.log`

Required for exact UI restore:

- `turns`
- `sessionAnnotations`

If `messages` is gone, continuation is impossible. If `turns` are gone, exact UI
restore is impossible, though a rough view could be reconstructed from messages.

## YAML Relationship

Current `JobConfig` mixes reusable agent recipe and invocation concerns:

Reusable agent-ish fields:

- `name`
- `provider`
- `tools`
- `provider_tools`
- `mcps`

Invocation/run fields:

- `task`
- `files`
- `batch`

Future shape could normalize to:

```ts
interface JobSpec {
  agent: AgentSpec;
  input: {
    task: string;
    files?: string[];
  };
  batch?: BatchConfig;
}
```

Existing YAML can remain accepted and normalize into this internal structure.

## Secrets Direction

Move away from provider secrets files. Prefer `.env` and environment variable
references.

Recommended YAML:

```yaml
provider:
  type: openai
  model: gpt-4.1
```

Resolver reads standard env vars:

- `OPENAI_API_KEY`
- `ANTHROPIC_API_KEY`
- `GEMINI_API_KEY`

Support explicit env references:

```yaml
provider:
  type: openai
  model: gpt-4.1
  apiKeyEnv: CUSTOM_OPENAI_KEY
```

Literal secrets in YAML should be discouraged or treated as an escape hatch,
not the happy path.

CLI should load `.env`. Core library should not load `.env`; library consumers
can choose dotenv, platform secrets, workers bindings, etc.

## Open Questions

- Should `AgentSpec` live in core or CLI?
- Should current `JobConfig` be normalized into `JobSpec` immediately or only
  after package split?
- Should built-in local tools stay in core, move to CLI, or move to a third
  package?
- Should MCP remain core-level or become CLI/host-level?
- What should `agent.snapshotState()` and `agent.restoreState()` look like?
- Should `Agent.fromSaved(saved, resolver)` exist as a convenience API, or only
  lower-level `new Agent(config)` plus `agent.restoreState(state)`?
