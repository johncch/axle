# Migrating to Axle 0.25.0

0.25.0 adds native-first web search fallback and provider-specific
Chat Completions compatibility:

- `configureAxle({ webSearchFallback })` configures a process-wide search
  backend
- `braveWebSearch()` provides the first bundled fallback backend using Brave
  Search's LLM Context endpoint
- `web_search` stays provider-native when the selected provider supports it
- providers without native search receive an ordinary executable
  `web_search` tool
- provider-tool name resolution now happens once before request conversion
- `chatCompletions()` gains `providerDialect: "together"` for Together's
  reasoning request shape
- Together PDF inputs fail locally with a clear unsupported-input error

The release is backward-compatible for applications that do not request
provider-managed web search. Applications that use
`providerTools: [{ type: "provider", name: "web_search" }]` with a generic
Chat Completions provider should configure a fallback or register their own
executable search tool.

---

## Native-First Web Search

Application code continues to request the portable provider tool:

```ts
const agent = new Agent({
  provider,
  model,
  providerTools: [{ type: "provider", name: "web_search" }],
});
```

Axle now resolves that request at the start of each `generate()`, `stream()`,
or `Agent.send()` operation:

1. If the provider supports native `web_search`, Axle sends the provider-native
   tool.
2. Otherwise, Axle exposes the configured fallback as an executable
   `web_search` tool.
3. If neither is available, Axle fails before sending the model request with
   `WEB_SEARCH_FALLBACK_NOT_CONFIGURED`.

OpenAI, Anthropic, Gemini, and OpenRouter configured with
`providerToolVendor: "openrouter"` retain their native search paths. A generic
Chat Completions provider has no native search mapping and therefore uses the
fallback.

### Configure Brave Search

Configure the fallback once during application startup:

```ts
import { braveWebSearch, configureAxle } from "@fifthrevision/axle";

configureAxle({
  webSearchFallback: braveWebSearch({
    apiKey: process.env.BRAVE_API_KEY!,
    maxResults: 5,
    maxTokens: 4_096,
    country: "US",
    searchLanguage: "en",
  }),
});
```

The backend calls `/v1/llm/context`, which searches and extracts passages
selected for relevance to the query. It does not use the traditional
`/v1/web/search` result descriptions.

Supported options:

```ts
interface BraveWebSearchOptions {
  apiKey: string;
  endpoint?: string;
  maxResults?: number; // final source URL limit, 1-50, default 5
  candidateCount?: number; // search candidates considered, 1-50
  maxTokens?: number; // total context budget, default 4,096, max 32,768
  maxSnippets?: number; // total passage count, max 256
  maxTokensPerUrl?: number; // max 8,192
  maxSnippetsPerUrl?: number; // max 100
  contextThresholdMode?: string;
  country?: string;
  searchLanguage?: string;
  freshness?: "pd" | "pw" | "pm" | "py" | `${string}to${string}`;
  timeoutMs?: number;
}
```

`maxResults` maps to Brave's `maximum_number_of_urls`. It limits the final
number of source pages rather than the number of search candidates considered.

`configureAxle()` is process-wide. Axle snapshots the current configuration
when an operation starts, so changing it does not alter an active run.

### Custom Fallback Backends

Brave is optional. Any `WebSearchBackend` can be installed globally:

```ts
import { configureAxle } from "@fifthrevision/axle";
import type { WebSearchBackend } from "@fifthrevision/axle";

const searchBackend: WebSearchBackend = {
  name: "internal-search",
  async search({ query }, { signal }) {
    const matches = await searchInternalIndex(query, { signal });
    return {
      results: matches.map((match) => ({
        title: match.title,
        url: match.url,
        snippets: match.passages,
      })),
    };
  },
};

configureAxle({ webSearchFallback: searchBackend });
```

Backends return normalized results with `title`, `url`, and `snippets`.
`snippets` preserves the boundaries between relevant passages selected from a
page:

```ts
interface WebSearchResult {
  title: string;
  url: string;
  snippets: string[];
}
```

Axle handles tool schema, execution events, and tool-result serialization.

### Native and Fallback Events Differ

Native provider search continues to emit provider-managed content and events:

- `provider-tool:start`
- `provider-tool:complete`
- provider-native citation parts or text citations

Fallback search is an ordinary Axle tool call:

- `tool:request`
- `tool:exec-start`
- `tool:exec-complete`
- a normal tool-result message containing normalized search results

Consumers should not require provider-tool events as proof that search ran.
Accept native citations/provider-tool activity or a successful `web_search`
tool result, depending on the selected provider.

### Provider Search Config

Provider-specific config remains available on native search:

```ts
providerTools: [
  {
    type: "provider",
    name: "web_search",
    config: { max_results: 5 },
  },
];
```

When fallback search is selected, this provider-specific `config` is ignored
and Axle emits a warning. Configure fallback behavior on `braveWebSearch()`
instead.

### Custom Search Behavior

Applications that want to own search execution should register an executable
`web_search` tool and omit the provider tool:

```ts
const webSearch = {
  name: "web_search",
  description: "Search the public web",
  schema: z.object({ query: z.string() }),
  async execute({ query }) {
    return await searchWithMyBackend(query);
  },
};

const agent = new Agent({
  provider,
  model,
  tools: [webSearch],
});
```

The global fallback only applies to the portable provider-tool request. It does
not replace an explicitly registered executable tool.

### Custom Providers

`AIProvider` now has an optional provider-tool resolver:

```ts
resolveProviderToolName?(name: string, model: string): string | undefined;
```

Returning a string declares native support and supplies the provider-native
name. Returning `undefined` declares the tool unsupported and allows
`web_search` fallback selection.

Custom providers that omit the method keep the previous passthrough behavior.
This avoids changing existing adapters that already interpret portable
provider-tool names internally.

## Together Chat Completions Dialect

Together's Chat Completions API differs from the generic OpenAI-compatible
shape for reasoning controls. Select the dialect when constructing the
provider:

```ts
import { chatCompletions } from "@fifthrevision/axle";

const together = chatCompletions("https://api.together.ai/v1", {
  apiKey: process.env.TOGETHER_API_KEY!,
  providerDialect: "together",
});
```

Normalized reasoning options are translated as follows:

| Axle option        | Generic Chat Completions   | Together dialect                |
| ------------------ | -------------------------- | ------------------------------- |
| `reasoning: true`  | `reasoning_effort: "high"` | `reasoning: { enabled: true }`  |
| `reasoning: false` | `reasoning_effort: "none"` | `reasoning: { enabled: false }` |
| omitted            | no reasoning field         | no reasoning field              |

Axle does not disable Together reasoning by default. Omitting `reasoning`
leaves the provider/model default unchanged. Raw `providerOptions` are still
applied last and can override normalized mappings.

### Together PDF Inputs

Together's Chat Completions API does not accept the OpenAI-compatible PDF file
part used by the generic adapter. With `providerDialect: "together"`, Axle now
rejects PDF inputs locally:

```text
Together Chat Completions does not support PDF file parts
```

Image support remains model-specific. A Together endpoint may accept image
parts while the selected model rejects multimodal input; qualify the exact
provider/model pair before relying on image workflows.

## Upgrade Checklist

1. If you use provider-managed `web_search` with generic Chat Completions,
   configure `webSearchFallback` or switch to an executable search tool.
2. Update event handling to accept ordinary tool events/results when fallback
   search is used.
3. Move fallback-specific search settings to `braveWebSearch()` rather than
   `ProviderTool.config`.
4. Add `providerDialect: "together"` to Together providers.
5. Do not assume file or multimodal support from the provider alone; validate
   the selected model.
