# Migrating to axle 0.13.0

0.13.0 is a breaking release focused on three features:

1. A first-class **tool registry** that lets tools mutate the agent's tool set at runtime (e.g. `load_tools` patterns).
2. **Streaming tool results** — tools can emit progress chunks via `ctx.emit`, surfaced as `tool:exec-delta` / `action:progress` events.
3. **Streaming tool args (client-side)** — the model's tool arguments surface as `tool:args-delta` / `action:args-delta` events so UIs can render args as they're generated. Tool execution lifecycle is unchanged.

The work also folded in a vocabulary cleanup that had been deferred. There's no back-compat shim layer; if you're upgrading you'll need to apply the changes below.

---

## TL;DR migration

```diff
- import { Agent, type ServerTool, type AxleTool } from "@fifthrevision/axle";
+ import { Agent, type ProviderTool } from "@fifthrevision/axle";

  const agent = new Agent({
    provider, model,
-   tools: [
-     calculatorTool,
-     { type: "server", name: "web_search" },
-   ],
+   tools: [calculatorTool],
+   providerTools: [{ type: "provider", name: "web_search" }],
  });

- agent.addTool(myTool);
- agent.tools["my-tool"];
+ agent.registry.add(myTool);
+ agent.registry.get("my-tool");
```

```diff
  const myTool: ExecutableTool = {
    name: "x", description: "...", schema,
-   async execute(input) {
+   async execute(input, ctx) {
      return "ok";
    },
  };
```

```diff
  // YAML job config
- server_tools:
+ provider_tools:
    - web_search
```

If you only consume Agent and built-in tools, that's everything. The rest of this document covers the full surface.

---

## Renames

The `Server` / `Internal` vocabulary was overloaded — "server" collided with MCP servers and HTTP servers, "internal" begged the question "internal to what." Everything that referred to provider-executed tools (web search, code interpreter, etc.) now consistently uses **`Provider`**.

| Before                                                         | After                                                                                             |
| -------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| `ServerTool` (type)                                            | `ProviderTool`                                                                                    |
| `{ type: "server", name }` (discriminator)                     | `{ type: "provider", name }`                                                                      |
| `ContentPartInternalTool`                                      | `ContentPartProviderTool`                                                                         |
| `"internal-tool"` (content part type)                          | `"provider-tool"`                                                                                 |
| `InternalToolAction` (turn action)                             | `ProviderToolAction`                                                                              |
| `"internal-tool"` (action kind)                                | `"provider-tool"`                                                                                 |
| `StreamEvent` `internal-tool:start` / `internal-tool:complete` | `provider-tool:start` / `provider-tool:complete`                                                  |
| Stream chunks `internal-tool-start` / `internal-tool-complete` | `provider-tool-start` / `provider-tool-complete`                                                  |
| YAML field `server_tools:`                                     | `provider_tools:`                                                                                 |
| Anthropic adapter helper `convertToProviderTools`              | `convertToAnthropicTools` (was internally misnamed; collided with the new `ProviderTool` concept) |

If you reference any of these names — in TypeScript imports, string discriminators, or YAML configs — apply the rename.

---

## `AxleTool` deleted

The `AxleTool = ExecutableTool | ServerTool` union is gone. The two kinds are now declared separately everywhere: a function you run locally (`ExecutableTool`) and a capability the provider runs (`ProviderTool`).

Where you used to declare them in one mixed array:

```diff
- const tools: AxleTool[] = [calculatorTool, { type: "server", name: "web_search" }];
+ const tools: ExecutableTool[] = [calculatorTool];
+ const providerTools: ProviderTool[] = [{ type: "provider", name: "web_search" }];
```

---

## Agent surface

`Agent` no longer exposes parallel `tools` / `providerTools` fields and `addTool*` methods. Everything routes through a single **`agent.registry`** of type `ToolRegistry`.

### Constructor

```diff
  new Agent({
    provider, model,
-   tools: [calculatorTool, { type: "server", name: "web_search" }],
+   tools: [calculatorTool],
+   providerTools: [{ type: "provider", name: "web_search" }],
  });
```

Both `tools` and `providerTools` are still optional; either or both may be omitted.

### Mutation / inspection

| Before                                                           | After                                 |
| ---------------------------------------------------------------- | ------------------------------------- |
| `agent.addTool(t)`                                               | `agent.registry.add(t)`               |
| `agent.addTools([t1, t2])`                                       | `agent.registry.add([t1, t2])`        |
| `agent.addProviderTool(t)` (didn't exist before — was `addTool`) | `agent.registry.addProvider(t)`       |
| `agent.tools["name"]`                                            | `agent.registry.get("name")`          |
| `agent.providerTools`                                            | `agent.registry.provider()`           |
| `Object.values(agent.tools)`                                     | `agent.registry.executable()`         |
| `agent.hasTools()`                                               | unchanged (now reads `registry.size`) |

`agent.registry` is `readonly` — you mutate via its methods, you don't replace it.

### Removal semantics changed

`ToolRegistry.add` **throws** on duplicate name (old `addTool` silently replaced). If you need to replace, `remove(name)` first.

```ts
agent.registry.remove("my-tool");
agent.registry.add(updatedTool);
```

This affects MCP re-resolution too — the agent now tracks per-MCP resolution (`WeakSet<MCP>`) so `addMcp()` after the first `send()` works without trying to re-add already-registered tools.

## `ExecutableTool.execute` signature change

Every executable tool now receives a **`ToolContext`** as its second argument. The first argument (validated input) is unchanged.

```ts
interface ToolContext {
  registry: ToolRegistry; // mutate the live tool set
  signal: AbortSignal; // cancellation
  emit: (chunk: string) => void; // stream progress chunks back
  tracer?: TracingContext; // per-tool span
}

interface ExecutableTool<TSchema> {
  // ...
  execute(input: z.infer<TSchema>, ctx: ToolContext): Promise<string | ToolResultPart[]>;
}
```

If your tool ignores `ctx`, the change is a no-op type-wise:

```diff
- async execute(input) {
+ async execute(input, _ctx) {
    // ...
  }
```

(For function-based tools — `execute: async (input) => { ... }` — TypeScript permits omitting the trailing parameter, so you may not need to change anything. For class-based tools implementing `ExecutableTool<...>`, the implementation must match the interface and you'll need the second parameter.)

### What `ctx` enables

```ts
async execute(input, ctx) {
  // 1. Cancellation
  if (ctx.signal.aborted) throw new Error("aborted");

  // 2. Live progress (streams to consumers)
  ctx.emit("started\n");

  // 3. Mutating the agent's tools mid-call — load_tools pattern
  ctx.registry.add(anotherTool);

  // 4. Nested tracing
  const span = ctx.tracer?.startSpan("substep");

  return "done";
}
```

## `stream()` and `generate()` options

The `tools` field changed shape and gained a `registry` alternative.

| Before                                                          | After                                         |
| --------------------------------------------------------------- | --------------------------------------------- |
| `tools?: ToolDefinition[]` (just `{name, description, schema}`) | `tools?: ExecutableTool[]` (the actual tools) |
| (no separate field)                                             | `providerTools?: ProviderTool[]`              |
| (no field)                                                      | `registry?: ToolRegistry`                     |

The three keys are mutually exclusive between `registry` and the `tools`/`providerTools` shortcuts. Using both throws `TOOL_OPTIONS_CONFLICT` at runtime.

If you used `stream()` directly with declaration-only tool objects:

```diff
- const myTool = { name: "x", description: "...", schema };
+ const myTool: ExecutableTool = {
+   name: "x", description: "...", schema,
+   async execute() { return "noop"; }, // unreachable when onToolCall handles execution
+ };

  stream({
    provider, model, messages,
    tools: [myTool],
    onToolCall: async (name, params, ctx) => { /* your impl */ },
  });
```

The stub `execute` is required by the new type, but `onToolCall` still wins — `tool.execute` is only invoked through Agent's onToolCall, not stream's directly.

If you want a registry instead (e.g. to share state across multiple `stream()` calls):

```ts
import { ToolRegistry } from "@fifthrevision/axle";

const registry = new ToolRegistry({ tools: [myTool] });
stream({ provider, model, messages, registry /* ... */ });
```

`generate()` follows the same shape.

## `ToolCallCallback` signature

`onToolCall` now receives a third argument — `ctx: ToolContext`.

```diff
- onToolCall: async (name, parameters) => {
+ onToolCall: async (name, parameters, ctx) => {
    // your dispatch logic
  }
```

The `ctx` is the same one your tools' `execute` receives. If you're forwarding to a tool, pass it through.

## `ActionResult` gains `in-progress`

The discriminated union for action results now has three states instead of two:

```ts
type ActionResult =
  | { type: "in-progress"; content: string } // NEW — set during streaming
  | { type: "success"; content: unknown }
  | { type: "error"; error: { type: string; message: string } };
```

`ToolAction.detail.result` evolves through these states as a streaming tool runs:

- `undefined` — tool hasn't started
- `{ type: "in-progress", content: "" }` — first delta arrived
- `{ type: "in-progress", content: accumulated }` — chunks accumulating
- `{ type: "success", content }` or `{ type: "error", error }` — finished

If your code narrows on `result.type`, widen it accordingly:

```diff
- if (result.type === "success") render(result.content);
+ if (result.type === "success" || result.type === "in-progress") render(result.content);
```

`SubagentAction.detail.result` shares the same `ActionResult` type but in practice never goes through the `in-progress` state (sub-agents are atomic from the parent's view).

## New event types

The tool-call lifecycle now emits two new pairs of events. Each pair has a
StreamEvent (low-level, fired by `stream()`) and a corresponding AgentEvent
(translated by the turn builder, what `agent.on()` callbacks see).

Chronologically: `tool:args-delta` arrives as the model streams its tool
arguments → `tool:exec-start` (existing) → `tool:exec-delta` arrives if the
tool calls `ctx.emit(...)` during execution → `tool:exec-complete` (existing).

### Args streaming — `tool:args-delta` / `action:args-delta`

```ts
// StreamEvent
{
  type: "tool:args-delta";
  index: number;
  id: string;
  name: string;
  delta: string;
  accumulated: string;
}

// AgentEvent
{
  type: "action:args-delta";
  turnId: string;
  partId: string;
  delta: string;
  accumulated: string;
}
```

Fires as the model generates a tool call's arguments. Carries **raw partial
JSON** — consumers can display it as-is for "tool is being typed" UIs, or
run their own partial-JSON parser (e.g. `partial-json` npm) for structured
rendering. The framework doesn't attempt to parse partials.

Tool execution lifecycle is unchanged: `execute(input, ctx)` still runs once
after `tool-call-complete` with fully-validated, fully-typed params.

While args stream in, `ToolAction.detail.pendingArgs` (a new `string` field)
accumulates the raw partial JSON. On `tool:exec-start`, `pendingArgs` is
cleared and `parameters` is set with the parsed value.

Gemini synthesizes a single args-delta with the full JSON since its API
delivers args atomically rather than incrementally — this keeps the consumer
event surface uniform across providers.

### Result streaming — `tool:exec-delta` / `action:progress`

```ts
// StreamEvent
{
  type: "tool:exec-delta";
  index: number;
  id: string;
  name: string;
  chunk: string;
}

// AgentEvent
{
  type: "action:progress";
  turnId: string;
  partId: string;
  chunk: string;
}
```

Fires when a running tool calls `ctx.emit(chunk)`. Symmetric with `text:delta`
for assistant text. Subscribe via `agent.on()` to render live tool progress:

```ts
agent.on((event) => {
  if (event.type === "action:progress") {
    process.stdout.write(event.chunk);
  }
});
```

Each chunk is also accumulated onto `ToolAction.detail.result` as
`{ type: "in-progress", content: accumulated }` (see [`ActionResult` gains
`in-progress`](#actionresult-gains-in-progress) above).

## Built-in tools

`execTool` now actually streams. Long-running shell commands surface stdout/stderr as `action:progress` events while they run, and `ctx.signal.abort()` kills the subprocess.

`runCommand` (the helper in `src/tools/exec/helpers.ts`) gained two options:

```ts
interface ExecOptions {
  cwd?: string;
  timeout?: number;
  maxBuffer?: number;
  signal?: AbortSignal; // NEW
  onChunk?: (chunk: string) => void; // NEW
}
```

Existing call sites that don't pass these are unaffected.

The other built-in tools (`calculatorTool`, `readFileTool`, `writeFileTool`, `patchFileTool`, `braveSearchTool`) accept the new `ctx` parameter but don't yet use it.

## Top-level exports

These were documented in the README as public surface but weren't actually exported from the entry point. Now they are:

- `execTool`
- `readFileTool`
- `writeFileTool`
- `patchFileTool`
- `ToolRegistry` (class)
- `ToolContext` (type)

If you were importing them from `@fifthrevision/axle/tools/...`, you can now import directly from `@fifthrevision/axle`.

---

## What did NOT change

- **Provider API** (`anthropic()`, `openai()`, `gemini()`, `chatCompletions()`) — unchanged.
- **`Instruct`** — unchanged.
- **History / messages** — unchanged shape (apart from the `internal-tool` → `provider-tool` rename in content part discriminators).
- **Memory** (`ProceduralMemory`, `AgentMemory` interface) — unchanged.
- **Tool result format** (`ToolResultPart`, `AxleToolCallResult`) — unchanged.
- **Tool execution lifecycle** — unchanged. `execute(input, ctx)` still runs once after `tool-call-complete` with fully-validated, fully-typed params. The new `tool:args-delta` / `action:args-delta` events are client-side visibility only (let UIs render the model "typing"). Streaming model output _into_ a running tool (e.g. `write-file` writing bytes as the model generates content) is a 0.14+ feature.

---

## What's next

Likely 0.14 directions, in case you want to plan around them:

- **Streaming params _into_ tools** — let tools start running before the assistant turn closes (writing partial content to disk as the model generates it, etc.).
- **TTL / scope-based tool eviction** — only if `load_tools` patterns warrant it; manual `unload_tools` is the v1 escape hatch.
- **Structured `ctx.emit` chunks** — currently `string`; may generalize to typed deltas (e.g. stdout vs stderr labels for `exec`).
