# Migrating to Axle 0.24.0

0.24.0 adds subagent tools and tool fan-out, and makes token usage
reconstructable across models:

- `createAgentTool` exposes a child `Agent` as a normal tool (experimental)
- `parallelize` wraps a tool in a concurrent batch variant (experimental)
- `Stats` gains a flat per-provider/model `breakdown` for cost reconstruction
  (experimental)
- `ToolContext` gains `reportUsage`, and `emit` accepts structured chunks
- new stream/turn events for subagent progress and terminal tool errors
  (experimental)
- `Instruct.addContext` appends host-supplied supporting context without
  modifying the authored prompt
- tool results can return deferred file references and resolve them at each
  provider request
- **behavior change:** `stream()` `onToolCall` returning `null` now falls
  through to the registry tool
- **behavior change:** a tool-internal `AbortError` no longer aborts the run
  while the run's signal is live
- **render change:** `Instruct` text references now use valid, collision-safe
  Markdown fences

The features marked experimental are usable today, but their shapes —
`SubagentAction.detail`, `UsageEntry`, `ParallelToolResult`, and the new event
payloads — may change in a minor release while they are validated in real
applications. The two behavior changes are **not** experimental: they apply to
every `stream()` user unconditionally.

Most applications need changes only if they pass `onToolCall` to `stream()`,
construct `ToolContext` objects by hand, switch exhaustively over
`StreamEvent`/`TurnEvent`, persist `Stats` objects, or snapshot the exact output
of `Instruct.render()`.

## `Instruct.addContext`: Host-Supplied Supporting Context

Hosts can now append ordered context sections without concatenating them into
the user-authored prompt:

```ts
const instruct = new Instruct({
  prompt: "Inspect the current sandbox and suggest the next change.",
});

instruct.addContext("src/index.ts\npackage.json", {
  title: "Sandbox files",
});
instruct.addContext("Node.js 24\nPackage manager: pnpm", {
  title: "Environment",
});
```

Context sections:

- are stored separately from `prompt`
- survive `clone()`, `withInput()`, and `withInputs()`
- render in insertion order after text references
- do not perform `{{variable}}` substitution
- become part of the same final user-message text as the prompt

The last point matters for trust boundaries: `addContext` provides composition
and ownership semantics, not a separate model instruction priority.

No migration is required. This is an additive alternative to manually
concatenating manifests, environment details, retrieved context, or other
host-owned material into `prompt`.

### Text-reference rendering change

Text references and context sections now place content on a new line inside a
Markdown fence. The fence grows beyond the longest backtick run in the content,
so embedded triple-backtick code blocks cannot close it early.

Applications that compare the exact output of `Instruct.render()` should update
their snapshots. The model-visible content is otherwise unchanged.

## Deferred Files in Tool Results

`ToolResultPart.file` now accepts `FileInfo` instead of only
`ConcreteFileInfo`. A tool can therefore return the same deferred
`{ type: "ref" }` source accepted by user-message file parts:

```ts
const readFile = {
  name: "read_file",
  description: "Read a file from the sandbox",
  schema: z.object({ id: z.string() }),
  async execute({ id }) {
    return [
      {
        type: "file" as const,
        file: {
          kind: "image" as const,
          mimeType: "image/png",
          name: "chart.png",
          source: { type: "ref" as const, ref: { id } },
        },
      },
    ];
  },
};

const agent = new Agent({
  provider,
  model,
  tools: [readFile],
  fileResolver: async ({ ref, accepted }) => {
    return mintFileSource(ref, accepted);
  },
});
```

Resolution remains a provider-boundary operation. The authored `ref` stays in
message history, turns, snapshots, and traces; it is resolved again whenever a
later provider request re-sends that tool result. This is intentional for
short-lived signed URLs, which should not be persisted into history.

For persisted sessions, the host-owned `ref` value should be JSON-serializable.
The host must also restore a compatible `FileResolver` when continuing the
session.

Provider support is unchanged:

- Anthropic, OpenAI Responses, and Gemini support deferred image, PDF, and text
  tool-result files within their existing file-format constraints
- Chat Completions resolves deferred text tool-result files, but still rejects
  image and document tool-result parts

Existing tools returning inline text, base64 data, or URLs need no changes.

## `stream()`: `onToolCall` Returning `null` Executes the Registry Tool

Previously `stream()` and `generate()` disagreed about what a user-provided
`onToolCall` returning `null`/`undefined` meant: `generate()` fell through to
executing the matching registry tool, while `stream()` produced a `not-found`
error without executing anything. Both now fall through.

If you used a `null` return as a permission gate in `stream()`, return an
explicit error result instead.

### Before

```ts
stream({
  provider,
  model,
  messages,
  tools: [writeFile],
  onToolCall: async (name) => {
    if (!allowed.has(name)) return null; // relied on: tool never ran
    return undefined;
  },
});
```

### After

```ts
stream({
  provider,
  model,
  messages,
  tools: [writeFile],
  onToolCall: async (name) => {
    if (!allowed.has(name)) {
      return { type: "error", error: { type: "denied", message: `Tool not allowed: ${name}` } };
    }
    return undefined; // fall through to the registry tool
  },
});
```

### Why

`onToolCall` is an interceptor: `null` means "I don't handle this call," and
the engine resolves it like any other call. One set of semantics across
`stream()` and `generate()` also lets `Agent` stop shipping its own execution
override.

## Tool-Internal `AbortError`s Are Recoverable While the Run Is Live

A tool that throws an error merely _named_ `AbortError` — typically a
`DOMException` from the tool's own `fetch` timeout or internal
`AbortController` — no longer terminates the whole send when the run's signal
has not aborted. The model now receives an ordinary error tool result and can
retry or continue.

Termination still happens for: `AxleToolFatalError`, `AxleAbortError`, or any
error thrown while the run's own `signal` is aborted.

### Before

```ts
// tool internals
const inner = AbortSignal.timeout(5_000);
await fetch(url, { signal: inner }); // timeout → DOMException "AbortError"
// → entire agent.send() rejected as aborted, reason undefined
```

### After

The same timeout produces a model-visible tool error:

```json
{ "error": { "type": "execution", "message": "The operation was aborted." } }
```

If a tool genuinely needs to stop the run, throw `AxleAbortError` (cancel) or
`AxleToolFatalError` (unrecoverable failure) explicitly.

## `ToolContext`: Structured `emit` and Optional `reportUsage`

```ts
export interface ToolContext {
  registry: ToolRegistry;
  signal: AbortSignal;
  emit: (chunk: ToolProgressChunk) => void; // was (chunk: string) => void
  reportUsage?: (usage: Stats) => void; // new
  span?: Span;
}
```

`ToolProgressChunk` is `string | { type: "turn-event"; event: TurnEvent }`.
Tool authors are unaffected — `ctx.emit("progress…")` still typechecks, and
`reportUsage` is optional to call.

Code that _constructs_ a `ToolContext` by hand (test harnesses, direct
`tool.execute` callers) must widen its `emit` parameter type. Handlers that
only understand strings can ignore structured chunks:

### Before

```ts
const ctx: ToolContext = { registry, signal, emit: (chunk: string) => log(chunk) };
```

### After

```ts
const ctx: ToolContext = {
  registry,
  signal,
  emit: (chunk) => {
    if (typeof chunk === "string") log(chunk);
  },
};
```

Tools that spend model tokens (subagents, LLM-backed tools) should call
`ctx.reportUsage?.(usage)` so their spend rolls into the parent operation's
totals.

## `Stats` Gains a Flat `breakdown`

`Stats` now extends `TokenStats` and may carry attribution entries:

```ts
interface UsageEntry extends TokenStats {
  provider: string;
  model: string;
}

interface Stats extends TokenStats {
  breakdown?: UsageEntry[]; // one entry per provider+model pair
}
```

Every model turn is attributed, and `addStats`/`mergeStats` merge entries by
provider+model key, so `result.usage.breakdown` always sums exactly to the
aggregate fields. Cost is reconstructable even when subagents run on different
providers or models:

```ts
const cost = result.usage.breakdown?.reduce((total, e) => total + price(e.provider, e.model, e), 0);
```

Notes for existing consumers:

- breakdown entries **explain** the totals; never add them to the totals again
- persisted/serialized `Stats` objects grow a `breakdown` array
- `TokenStats` and `UsageEntry` are exported; `mergeStats` joins `addStats` and
  `createStats` in the public API

## New Events for Subagents and Terminal Tool Errors

`StreamEvent` additions:

- `tool:request` carries `kind?: "tool" | "agent"`
- `tool:exec-delta` chunks are `ToolProgressChunk` (child turn events arrive as
  `{ type: "turn-event", event }`)
- `tool:exec-complete` carries `usage?: Stats`
- new `tool:exec-error` event for fatal/aborted tool calls

`TurnEvent`/part additions:

- `SubagentAction` part (`kind: "agent"`) whose `detail.children` accumulates
  the child agent's turns
- `action:child-event` routes a child turn event to its agent part
- `action:error` finalizes a part for fatal/aborted calls

Consumers with exhaustive `switch` statements over these unions need new
cases; renderers that ignore unknown events keep working. A tool call that
ends in `tool:exec-error` does **not** also emit `tool:exec-complete` — treat
either as terminal when tracking pending calls.

## `Agent.on()` Returns an Unsubscribe Function

```ts
const unsubscribe = agent.on(handler);
// …
unsubscribe();
```

Previously the return type was `void`. Purely additive, but long-lived agents
that attach per-operation listeners should now detach them.

## New: `createAgentTool` and `parallelize`

Not a migration item — see the README sections "Subagent Tools",
"Parallelizing Tools", and "Usage Stats" for the new APIs. The short version:

```ts
const researcher = createAgentTool({
  name: "research",
  description: "Delegate a research question to a focused subagent",
  schema: z.object({ question: z.string() }),
  createAgent: () => new Agent({ provider, model: cheapModel, system: "…" }),
  prompt: (input) => input.question,
});

const agent = new Agent({
  provider,
  model,
  tools: [parallelize(researcher, { maxConcurrency: 4 })],
});
```

Child turn events stream through the parent (`action:child-event`), child
tokens land in `usage.breakdown`, and fatal/abort semantics match unbatched
tools.
